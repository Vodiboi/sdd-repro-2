# A5000 Training Handoff

This folder contains the server-side training workflow for the SSD records
generated by the parent project. The scripts accept a single JSONL file, an
output directory from `generate_ssd_dataset.py`, or a folder of split
`sft_messages_*.jsonl` chunks.

## Files

- `train_qwen_lora.py` - Unsloth LoRA fine-tuning entrypoint.
- `merge_qwen_lora.py` - optional post-training merge/export entrypoint.
- `validate_training_data.py` - dependency-free JSONL sanity check.
- `prepare_code_only_sft.py` - strips thinking/prose/fences from generated
  records and writes code-only SFT chunks.
- `export_training_metrics.py` - export loss history from checkpoints.
- `requirements-a5000.txt` - minimal Python packages to install in a CUDA venv.

## Data Input

The trainer accepts both full records and SFT-only records. Each line must have:

```json
{"messages": [{"role": "system", "content": "..."}, {"role": "user", "content": "..."}, {"role": "assistant", "content": "..."}]}
```

If `--data-path` is a directory, the trainer prefers files in this order:

1. `sft_messages/sft_messages_*.jsonl`
2. direct `sft_messages_*.jsonl` files
3. `sft_messages.jsonl`
4. `records/records_*.jsonl`
5. direct `records_*.jsonl` files
6. `records.jsonl`
7. any other direct `*.jsonl` files except `skipped_rows.jsonl`

Pass `--prefer-records` to reverse that preference. If your final handoff is a
single local `records.jsonl`, just pass its path directly.

## Setup On The A5000

Use a fresh Linux CUDA environment. The exact PyTorch/CUDA wheel can depend on
the server image, so check the installed driver with `nvidia-smi` first.

```bash
python3.10 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r training/requirements-a5000.txt
```

Keep Hugging Face credentials out of notebooks and scripts:

```bash
export HF_TOKEN=...
```

## Prepare Code-Only Data

The raw generated sample originally contained Qwen thinking traces and prose.
Do not train on that directly for LiveCodeBench accuracy. Build code-only
targets first:

```bash
python training/prepare_code_only_sft.py \
  --data-path training/sft_messages \
  --output-dir training/sft_code_only
```

This writes chunks under `training/sft_code_only/sft_messages/`, replaces the
system prompt with a code-only instruction, extracts the final Python code block,
and skips records that still look like prose after cleaning.

## Validate Data

From the repository root:

```bash
python training/validate_training_data.py \
  --data-path training/sft_code_only
```

For the future single-file handoff:

```bash
python training/validate_training_data.py \
  --data-path /path/to/records.jsonl
```

## Smoke Test

This runs on only a few local records and writes a disposable adapter folder.

```bash
python training/train_qwen_lora.py \
  --data-path training/sft_code_only \
  --limit-records 3 \
  --max-steps 2 \
  --save-steps 1 \
  --output-dir training_outputs/smoke_qwen3_4b_lora
```

## Full Training Run

The defaults use Qwen3-4B, 4-bit loading, LoRA rank 32, sequence length 2048,
batch size 2, gradient accumulation 4, response-only training, a conservative
LoRA learning rate of `2e-5`, and no automatic Hub push. For serious runs, pass
the context length explicitly instead of relying on defaults.

```bash
python training/train_qwen_lora.py \
  --data-path training/sft_code_only \
  --output-dir training_outputs/qwen3_4b_lora \
  --max-steps -1 \
  --num-train-epochs 1 \
  --learning-rate 2e-5
```

The raw split `training/sft_messages` has 1005 records, but it contains thinking
traces. The prepared split `training/sft_code_only` has 979 code-only records.
Use `validate_training_data.py` to inspect lengths before picking a context
length. On the A5000, start with a smoke test, then try a larger context with
smaller batch size if VRAM allows:

```bash
python training/train_qwen_lora.py \
  --data-path training/sft_code_only \
  --max-seq-length 32768 \
  --per-device-train-batch-size 1 \
  --gradient-accumulation-steps 8 \
  --learning-rate 2e-5 \
  --max-steps -1 \
  --num-train-epochs 1 \
  --output-dir training_outputs/qwen3_4b_lora
```

For a longer run, either raise `--max-steps` or set it to `-1` and use
`--num-train-epochs`. Run it under `tmux`, `screen`, or your scheduler rather
than a notebook kernel.

Resume from a saved checkpoint:

```bash
python training/train_qwen_lora.py \
  --data-path /path/to/records.jsonl \
  --output-dir training_outputs/qwen3_4b_lora \
  --resume-from-checkpoint training_outputs/qwen3_4b_lora/checkpoint-25
```

## Metrics

Trainer checkpoints include `trainer_state.json`, which stores `log_history`
entries such as loss, learning rate, gradient norm, step, and epoch. Export them
without touching the training tmux session:

```bash
python training/export_training_metrics.py \
  --output-dir training_outputs/qwen3_4b_lora
```

The exporter writes:

```text
training_outputs/qwen3_4b_lora/metrics/training_metrics.csv
training_outputs/qwen3_4b_lora/metrics/loss.png
```

## Merge Or Export

Only do this after the LoRA adapter is saved and inspected.

```bash
python training/merge_qwen_lora.py \
  --adapter-dir training_outputs/qwen3_4b_lora \
  --output-dir training_outputs/qwen3_4b_merged_16bit \
  --export merged_16bit
```

Pushes are opt-in:

```bash
python training/train_qwen_lora.py \
  --data-path /path/to/records.jsonl \
  --output-dir training_outputs/qwen3_4b_lora \
  --push-to-hub \
  --hub-model-id your-username/qwen3_4b_lora
```
